## 1. What this project is

A single Cloudflare Worker that ingests a daily Excel workbook (Air Force
base vehicle fleet status) and serves a multi-tab dashboard plus a mobile
walker app on top of it. Three distinct UIs live behind one Worker:

| Surface | URL | Audience |
|---|---|---|
| **Desktop dashboard** | `/` | Fleet manager / dispatcher at a desk |
| **Walker (mobile)** | `/yard` | Walker out in the lot with a phone |
| **Mechanic (mobile)** | `/waivers` | Mechanic at the bay submitting/verifying waivers |
| **Waiver card (print)** | `/waivers/card/<assetId>/print` | Printable card that lives in the cab |
| **Presenter (TV)** | `/?present=<meetingId>` | Conference-room screen during ETIC standup |

The **same `fetch` handler** in `src/index.ts` serves all three, plus all
JSON APIs.

### Domain glossary

- **ETIC** ‚Äî Estimated Time In Commission. Each broken vehicle has an ETIC
  date by which it's expected back in service. The whole app exists to
  manage these.
- **WO / Work Order** ‚Äî One repair job tied to one asset. Has an ETIC date,
  a remarks field, a MEL tier.
- **MEL** ‚Äî Master Equipment List. A *category* of vehicles (e.g. "M16/MEL key
  ARMOR-91-MSOSO"). Each MEL has a required minimum FMC count; the rollup
  shows whether the unit is **Above / At / Below** that threshold.
- **MEL tier** for a WO ‚Äî `above` / `at` / `below`. Below = the WO is
  contributing to a MEL shortage; the most painful kind.
- **NCE** ‚Äî Nuclear Certified Equipment. Special handling.
- **FMC / NMC** ‚Äî Fully / Non-Mission Capable.
- **FM&A** ‚Äî the maintenance follow-up queue (Findings, Maintenance & Action).
- **Yard check / sighting** ‚Äî A walker physically tapping "I see this asset
  here at lot B." Used to compute "last seen" badges across the app.
- **Waiver** ‚Äî A defect on a vehicle that management has formally accepted
  (doesn't affect safety/serviceability). Mechanics submit a request from
  `/waivers` with photo + description; management approves on the desktop.
  Approved waivers appear on a printable card kept in the vehicle and must
  be re-verified at least annually.

## 2. Stack at a glance

- **Runtime**: Cloudflare Workers, ES2022 + WebWorker libs, `nodejs_compat`.
- **Persistence**: D1 (`ETIC_SNAPSHOTS`) for relational; R2 (`ETIC_BUCKET`)
  for raw workbooks + JSON analyses.
- **Trigger**: Email Routing ‚Üí `email()` handler ingests new workbooks.
- **Frontend**: **NO build step.** The Worker returns HTML strings. The
  desktop UI's HTML, CSS, and JS all live inside one giant TypeScript
  template literal in `src/index.ts`. See ¬ß4.
- **Tests**: Vitest, ~3 small unit-test files under `test/`.
- **CI**: `.github/workflows/ci.yml` runs `wrangler types` + `tsc --noEmit`
  + `vitest` on every push/PR to `main`.

## 4. The `src/index.ts` elephant

`src/index.ts` is **~14,000 lines** because the entire desktop dashboard's
HTML, CSS, and client-side JavaScript are embedded inside one TypeScript
template literal returned from `renderDashboardHtml()`. A second smaller
template literal in `renderYardAppHtml()` holds the walker app.

**This is intentional** ‚Äî it ships as one Worker bundle with no build step
or static asset pipeline. Don't try to extract it into separate files
without buy-in; the costs of a build pipeline outweigh the benefits here.

### Trap #1 ‚Äî backticks inside comments break the file

The whole template literal is delimited by backticks. **A stray backtick
inside a comment closes the literal mid-file** and produces a confusing
"Expected ';' but found 'someIdentifier'" error from esbuild during
`wrangler deploy`.

```ts
// BAD: this comment lives inside the template literal and the backticks
// in `escapeHtml` will close the surrounding literal.
const x = `<div>...`;

// GOOD: same comment, no backticks.
// The escapeHtml helper lives elsewhere; aliased here.
const x = `<div>...`;
```

**Rule:** Inside the dashboard/walker template literals, never use backticks
in comments. Use single quotes or just leave them out.

### Trap #2 ‚Äî the dashboard JS lives in **one big `<script>` block**

All of the dashboard's client-side code is in one `<script>` tag near the
bottom of `renderDashboardHtml()`. Function declarations there **hoist to
the top of that block**, so two functions with the same name silently
overwrite each other.

This bit us hard with `yardWireOnce()` ‚Äî there were two declarations, one
for the desktop FM&A panel and a stale one from the walker app's earlier
in-page version. The walker version hoisted last and silently broke the
desktop subnav clicks. **Fix**: rename one (we used `yardFmWireOnce`) and
update its single call site.

**Rule:** When adding a new top-level function in the dashboard `<script>`,
grep for the name first.

### Trap #3 ‚Äî multiple JS scopes, easy to miss

There are at least three independent script scopes in this file:

1. Desktop dashboard `<script>` block (huge).
2. Walker app `<script>` block inside `renderYardAppHtml()`.
3. Server-side TS helpers around them.

Functions/constants don't cross between them. The walker has its own
`escapeHtml()`; the desktop only has `esc()`. We aliased
`const escapeHtml = esc;` near the top of the desktop script so code copied
from the walker doesn't `ReferenceError`.

**Rule:** Before calling a helper inside the dashboard `<script>`, confirm
it's actually defined in *that* scope, not the walker's.

### Trap #4 ‚Äî large file = slow tools

Reading the whole file is expensive. Use `Grep` with line numbers + small
`Read` windows. There's a section map you can lean on:

| Approx lines | Section |
|---|---|
| 1‚Äì155 | Imports, types |
| 159‚Äì245 | `email()` handler (workbook ingest) |
| 247‚Äì460 | `fetch()` handler (route table) |
| 460‚Äì1340 | Server-side route handlers (`handleWatchApi`, `handleMelApi`, ‚Ä¶) |
| 2400‚Äì3500 | `renderYardAppHtml()` ‚Äî the walker app HTML+CSS+JS |
| 4400‚Äì7900 | Desktop dashboard CSS |
| 7900‚Äì8160 | Desktop dashboard HTML scaffold |
| 8160‚Äì14900 | Desktop dashboard `<script>` block |

(Line numbers drift as edits land ‚Äî use `Grep` to confirm before assuming.)

## 5. Data model ‚Äî D1 tables

Migrations are the source of truth (`migrations/00NN_*.sql`). High-level
map of the tables you'll interact with most often:

### `etic_snapshots`
One row per workbook ingest day. KPIs + R2 keys. Powers the home tab.

### `work_order_state` (and `work_order_history`)
Current and historical state of every WO ever seen. Day-over-day diffs feed
the WO change timeline and "stale remarks" detection. See `workOrderWatch.ts`.

### `mel_state`
Per-MEL-key fleet state. Powers the MEL tab. See `melWatch.ts`.

### `mel_config`, `mel_subdivision`
User-edited critical thresholds and MEL subdivisions. Settings UI writes here.

### `yard_check`
Every walker tap on an asset. Columns: `asset_id`, `location`,
`discrepancies`, `status`, `checked_by`, `checked_at_iso`, `source_date_key`.
**Key invariant:** only `status='present'` rows count as "sightings" for the
last-seen badge (see ¬ß7).

### `yard_finding_action`
FM&A "I dealt with this" log entries ‚Äî opened a WO, marked acknowledged,
re-opened a previously-closed finding, etc. Findings are computed live;
this table only records *resolutions*.

### `meeting`, `meeting_note`
ETIC standup sessions and per-WO discussion notes. The presenter polls
`/api/meeting/:id` every 2s.

### `waiver`, `waiver_verification`
Waiver card system (migration `0018_waivers.sql`). `waiver` rows are the
defect submissions (status: `pending` / `approved` / `rejected`) with one
optional R2 photo (`photo_r2_key` under `waiver-photos/<id>/...`).
`waiver_verification` is the immutable audit log ‚Äî every approval seeds
an `initial` row, then mechanics or management append `annual` / `adhoc`
rows from either app. The `last_verified_*` columns on `waiver` are
denormalized so list views can compute "overdue annual?" without a JOIN.
See `src/waivers.ts` and ¬ß9 "Add a new waiver-card style cross-cutting badge."

## 6. R2 layout

```
workbooks/<YYYY-MM-DD>/<file>.xlsx   raw inbound workbooks
analyses/<YYYY-MM-DD>.json           per-day analysis JSON (KPIs, sheet stats)
analyses/latest.json                 mirror of the most recent analysis
history/index.json                   rolling KPI series + per-day deltas
photos/yard/<assetId>/<id>.jpg       walker-uploaded photos
```

R2 is treated as the source of truth for the raw workbook + the canonical
analysis JSON. D1 holds the queryable, day-over-day relational view derived
from those workbooks.

## 7. Yard check semantics (important ‚Äî easy to get wrong)

The walker UI **does not** mark assets "missing." Walkers don't know what
*should* be on the lot, only what they see in front of them. Therefore:

- A walker tap *only ever* records `status='present'` for "I see it here."
  The legacy `missing` / `unknown` statuses still exist in the schema and in
  old rows, but new code never creates them.
- "**Missing**" findings in the FM&A queue are **absence-derived**: the
  rolling roster knows every asset that should exist. If the most recent
  `present` check for an asset is older than `2 √ó intervalDays`, it shows
  up as a "Not seen" finding. See `getMissingThresholdDays()` in
  `yardSession.ts`.
- "**Unlisted**" findings = a walker tapped an asset id that's not on the
  current roster.
- "**Discrepancy**" findings = walker noted something in `discrepancies`.
- The dashboard **last-sighting badge** (`renderSightingBadge`) shown next
  to every asset id across Work Orders / MEL / Meeting / Presenter is fed
  by `getLatestSightings()` ‚Üí `/api/yard/sightings`, which `GROUP BY`s on
  `asset_id` filtered to `status='present'` only.

When changing yard logic, **don't accidentally count non-`present` rows as
sightings.** That was a real bug.

## 9. Common workflows

### Add a new yard-check signal that should appear next to every asset id

This is the canonical "cross-cutting feature" pattern.

1. Server: extend `yardSession.ts` (or add a new helper) and a new
   `/api/yard/<thing>` endpoint in `src/index.ts`.
2. Client (desktop): add a small fetch + in-memory cache near
   `sightingsState` (top of the desktop `<script>` block).
3. Add a `renderXBadge(assetId)` helper that returns a small HTML pill.
4. Inject it at all six asset-rendering sites:
   - WO list cards (`renderWoList`, ~line 9300)
   - WO detail hero (`renderWoHero`, ~line 9490)
   - MEL ‚Üí Open WOs (`renderMelWosForKey`, ~line 12114)
   - Meeting queue (`renderMeetingQueue`, ~line 13800)
   - Meeting focus head (`renderMeetingFocus`, ~line 13880)
   - Presenter id strip (~line 14523)
5. Trigger your `loadX()` in the data-load entry points so the cache is
   warm: `loadAndRenderWoList`, `selectWo`, `renderMelWosForKey`,
   `enterLiveView`, `pollPresenter`.
6. Add a CSS class near other badge styles (around line 4990) and any
   presenter-specific size bump near `.p-id-strip` (around line 6892).

### Add a D1 column

1. New migration file `migrations/00NN_<name>.sql` (numeric, monotonic).
2. Apply locally: `npx wrangler d1 execute etic-snapshots --file=migrations/...`
3. Apply remotely: same command + `--remote`.
4. Update the read/write helpers in the appropriate `*.ts` file.
5. Regenerate types if needed: `npm run cf-typegen`.

### Debug a deploy failure

- `[ERROR] Expected ";" but found "..."` ‚Üí almost always a stray backtick
  inside a comment in the dashboard template literal. See trap #1.
- `Uncaught ReferenceError: foo is not defined` in browser ‚Üí the helper is
  defined in a *different* `<script>` scope (walker vs desktop). See trap #3.
- Subnav / button clicks silently doing nothing ‚Üí check for hoisted-name
  collision. See trap #2.

### Deploy

```powershell
npm run deploy
```

This runs `wrangler deploy --env-file .env.cloudflare`. The custom domain
`minot.2t3.app` updates within seconds. Always hard-refresh the browser
(Ctrl+Shift+R) ‚Äî the dashboard caches aggressively.

## 10. Cloudflare environment notes

`wrangler.jsonc` carries this deployment's:

- `account_id` ‚Äî Cloudflare account UUID
- `d1_databases[0].database_id` ‚Äî the D1 DB UUID
- `routes[0].pattern` ‚Äî `minot.2t3.app` custom domain

These are committed because they're not secrets and a single team owns this
deployment. **A fork** must replace all three (or `wrangler deploy` will
fail or, worse, push to the wrong account).

Secrets (`CLOUDFLARE_API_TOKEN`) live in `.env.cloudflare`, which is
gitignored. Template at `.env.cloudflare.example`.

## 12. Don't list

- ‚ùå Don't extract `src/index.ts` into separate files without explicit
  buy-in. The single-file shape is load-bearing.
- ‚ùå Don't add backticks to comments inside the dashboard/walker template
  literals.
- ‚ùå Don't redeclare a top-level dashboard `<script>` function name that
  already exists somewhere in the same script block.
- ‚ùå Don't introduce a build step (Vite / Rollup / etc.) without a strong
  case. The "no build step" property is what keeps deploys instant.
- ‚ùå Don't commit `debug-scrape.json` (gitignored). It contains real
  workbook rows.
- ‚ùå Don't remove the `worker-configuration.d.ts` from the repo without
  also removing it from `tsconfig.json` `include`. It's regenerated by
  `wrangler types` and the IDE wants it present for completion.
- ‚ùå Don't count non-`present` `yard_check` rows as "sightings."
- ‚ùå Don't have the walker UI mark assets as "missing." Missing is
  absence-derived from the roster (see ¬ß7).

## 13. When you're stuck

1. Read the imports and route table at the top of `src/index.ts` ‚Äî they
   double as a feature index.
2. The matching server-side logic lives in `workOrderWatch.ts`,
   `melWatch.ts`, `meeting.ts`, `yardSession.ts`, or `yardCheck.ts`.
3. For any UI puzzle, locate the `render*` function with `Grep` first; the
   dashboard JS is heavily compartmentalized into per-tab renderers.
4. If the browser shows stale UI after a deploy, hard-refresh ‚Äî Cloudflare
   edges and browsers both cache the HTML.
5. If `wrangler deploy` blows up with a TS parse error on a line that looks
   syntactically fine in your editor, it's almost certainly trap #1 or
   trap #2.
